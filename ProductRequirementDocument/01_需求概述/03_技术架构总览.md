## 3 数仓分层详解

### 3.1 五层模型定义

```
┌─────────────────────────────────────────────────────────────────┐
│ ADS  │ 应用数据层 (Application Data Store)                       │
│      │ 面向具体应用的高度汇总表、大宽表                            │
│      │ 形态：报表宽表、KPI 指标表、大屏数据集、监管报送视图        │
│      │ 保留：按应用需求（监管报表永久保留）                        │
├──────┼──────────────────────────────────────────────────────────┤
│ DWS  │ 汇总数据层 (Data Warehouse Summary)                        │
│      │ 按主题域轻度汇总，维度表 + 汇总事实表                        │
│      │ 形态：星型模型，日/周/月粒度快照                             │
│      │ 保留：5年在线 + 归档                                       │
├──────┼──────────────────────────────────────────────────────────┤
│ DWD  │ 明细数据层 (Data Warehouse Detail)                         │
│      │ ⭐ Data Vault 融合层                                       │
│      │ 形态：Hub(实体) + Link(关系) + Satellite(属性/事实)         │
│      │      + 传统拉链表 + 事务事实表                              │
│      │ 保留：3年在线 + 归档                                       │
├──────┼──────────────────────────────────────────────────────────┤
│ ODS  │ 操作数据层 (Operational Data Store)                        │
│      │ 贴源层，与源系统表结构 1:1 映射                              │
│      │ 增加：ingest_time(采集时间)、source_system(来源标识)        │
│      │ 保留：30天增量 + 每日全量快照                               │
├──────┼──────────────────────────────────────────────────────────┤
│ STG  │ 临时缓冲层 (Staging)                                       │
│      │ 原始落地数据，不做任何转换                                   │
│      │ 保留：3-7天，自动过期删除                                   │
└──────┴──────────────────────────────────────────────────────────┘
```

### 3.2 各层存储与分区策略

| 层 | 存储格式 | 压缩 | 分区键 | 分桶键 | 文件大小建议 |
|-----|---------|------|--------|--------|------------|
| STG | Text/CSV | 无 | `dt` (日期) | — | 不限制 |
| ODS | ORC | Snappy | `dt` + `source_system` | — | 128-256MB/文件 |
| DWD | ORC | Zlib | `dt` | `HASH_KEY` (bucket=32) | 128-256MB/文件 |
| DWS | ORC | Snappy | `dt` + `topic_domain` | `dim_key` (bucket=16) | 128-256MB/文件 |
| ADS | ORC/Parquet | Snappy | `dt` + `app_name` | — | 256-512MB/文件 |

### 3.4 命名规范

#### 表命名

```
格式: {layer}_{domain}_{entity_description}

layer:    stg / ods / dwd / dws / dim / ads
domain:   core / pay / loan / cc / ebank / cust / wealth / aml / gl / ecif
          dep / risk / reg / fin / party / product / channel / ib / ...

示例:
  stg_core_t_account          # STG 层-核心银行-账户表
  ods_core_t_account          # ODS 层-核心银行-账户表
  dwd_hub_customer            # DWD 层-客户 Hub
  dwd_link_cust_acct          # DWD 层-客户-账户 Link
  dwd_sat_customer_info       # DWD 层-客户信息 Satellite
  dwd_fact_transaction        # DWD 层-交易事务事实表
  dws_dep_daily_bal           # DWS 层-存款日余额汇总
  dim_customer                # 维度表-客户
  ads_mgmt_kpi                # ADS 层-管理驾驶舱 KPI
  ads_reg_1104_g01            # ADS 层-1104 报表 G01
```

#### 字段命名

```
规则:
  • 使用小写字母 + 下划线 (snake_case)
  • HASH_KEY 字段: {entity}_hash_key (如 customer_hash_key)
  • 业务主键: {entity}_id (如 customer_id)
  • 外键: 与被引用表字段名一致
  • 时间戳: {action}_time / {action}_date (如 trans_time, load_date)
  • 金额: {metric}_amt (如 trans_amt, balance_amt)
  • 标志位: is_{status} (如 is_active, is_deleted)
  • 来源标识: source_system, record_source

标准元数据字段 (所有表必须包含):
  • load_date     DATE        数据加载日期
  • record_source VARCHAR(50) 数据来源系统
  • etl_time      TIMESTAMP   ETL 处理时间
```

## 4 架构特色：Data Vault 与传统数仓的融合

### 4.1 为什么在 DWD 层引入 Data Vault

| 传统数仓 DWD 的痛点 | Data Vault 如何解决 |
|-------------------|-------------------|
| 多源系统主键不一致，人工对齐成本高 | HASH_KEY 自动生成，与源系统主键解耦 |
| 源系统表结构变更需要改 DWD 表 | 新增 Satellite 表即可，不影响现有结构 |
| 历史数据追溯需要复杂的拉链表 | Satellite 的 LOAD_DATE/LOAD_END_DATE 天然支持 SCD Type 2 |
| 审计溯源依赖于外部日志 | RECORD_SOURCE + LOAD_DATE 内置在每行数据中 |
| 多对多关系建模复杂 | Link 表天然建模多对多关系 |

### 4.2 融合策略

```
层            Data Vault 组件              传统数仓组件
────────────────────────────────────────────────────────
DWD           Hub (核心实体)               —
              Link (实体关系)              —
              Satellite (属性)            拉链表 (SCD Type 2)
              Satellite (事实)            事务事实表

DWS           —                          维度表 (Dim)
              —                          汇总事实表 (Fact)

ADS           —                          应用大宽表
```

### 4.3 Hash Key 生成算法

```sql
-- 使用 MD5 生成确定性 HASH_KEY
-- 确保跨系统、跨时间可复现

SELECT MD5(CONCAT(
    COALESCE(source_system, 'UNKNOWN'),  -- 来源系统
    '|',
    COALESCE(customer_id, ''),           -- 业务主键
    '|',
    'HUB_CUSTOMER'                        -- 实体类型(盐值)
)) AS customer_hash_key
FROM ods_core.t_customer;
```

## 5 安全与访问控制架构

| 层次 | 控制措施 | 实施方式 |
|------|---------|---------|
| **网络** | WSL2 本地环境，仅本机访问 | localhost 监听 |
| **HDFS** | 文件系统权限 + ACL | `hdfs dfs -chmod/chown` |
| **Hive** | 库/表级访问控制 | HiveServer2 Authorization |
| **数据脱敏** | DWD 层创建脱敏视图 | SQL VIEW + 脱敏 UDF |
| **审计** | HiveServer2 查询日志 | Audit Log to HDFS |

## 6 架构决策记录 (ADR)

| 编号 | 决策 | 理由 | 替代方案(已否决) |
|------|------|------|---------------|
| ADR-01 | 采用 Lambda 而非 Kappa 架构 | 银行业务对数据准确性要求极高，离线批处理可提供最终一致性保证 | Kappa：流批一体但银行案例偏少 |
| ADR-02 | DWD 层引入 Data Vault | 多源系统整合灵活，历史追溯天然支持，适合多源异构场景 | 纯星型模型：历史追溯需额外设计 |
| ADR-03 | Kafka 作为数据中枢而非旁路 | 统一实时和离线数据入口，避免 Lambda 双轨不一致 | Canal 直接写 HDFS：无法支撑实时消费 |
| ADR-04 | 使用 ORC 而非 Parquet | 与 Hive 集成更紧密，压缩比更高，支持 ACID (Hive 3.x) | Parquet：Spark 原生支持更好，但 Hive 集成不如 ORC |
| ADR-05 | 调度选择 DolphinScheduler | 分布式、DAG 可视化、社区活跃、中文文档完善 | Airflow：Python 依赖重；Oozie：XML 配置繁琐 |
