## 1 项目背景

### 1.1 银行业数据挑战

随着金融科技快速发展和监管趋严，商业银行面临以下数据挑战：

| 挑战 | 表现 | 影响 |
|------|------|------|
| **数据孤岛** | 核心银行、支付、信贷、理财等系统各自独立，缺乏统一数据视图 | 客户画像不完整、跨系统分析困难 |
| **监管压力** | 1104 非现场监管、EAST 标准化、反洗钱大集中等多套报送要求 | 数据口径不一致、报送成本高 |
| **实时需求** | 移动支付普及带来实时风控、实时对账需求 | 传统 T+1 批处理无法满足 |
| **数据质量** | 源系统数据标准不一、脏数据多 | 报表数据不可信、决策偏差 |
| **数据复用** | 各应用重复抽取加工，指标口径不统一 | 开发效率低、口径冲突 |

### 1.2 建设目标

本项目构建一个**完整的大型银行数据仓库**，解决上述挑战，实现以下目标：

| 目标 | 描述 | 衡量标准 |
|------|------|---------|
| **数据统一** | 整合 11 大源系统，建立统一数据视图 | 覆盖全行 13 大主题域 |
| **分层架构** | STG→ODS→DWD→DWS→ADS 五层模型，数据逐层加工 | 各层职责清晰、复用率高 |
| **离线+实时** | Lambda 架构：离线批处理(T+1) + 实时流处理(秒级) | 风控延迟<500ms，日批08:00前完成 |
| **数据治理** | 元数据管理、数据质量、数据血缘、数据安全全覆盖 | 关键表质量检查覆盖率 100% |
| **应用支撑** | 管理驾驶舱、监管报送、客户画像、风控、BI 五大应用 | 直接产出业务价值 |

## 3 设计原则

| 原则 | 说明 | 实践体现 |
|------|------|---------|
| **分层解耦** | 每层职责单一，下层为上层服务 | 五层模型，层间接口标准化 |
| **数据驱动** | 以数据质量为基础，元数据为驱动 | DQ 门禁在 DWS→ADS 前强制检查 |
| **渐进演进** | 先核心后边缘，先离线后实时 | 三期实施路线：核心域→全业务→实时增强 |
| **内外融合** | 国内五层架构 + 外企 Data Vault | DWD 层以 Hub-Link-Sat 组织 |
| **技术中立** | 设计不绑定特定版本，标注适配范围 | 各组件标注版本号与兼容性 |
| **可落地** | 所有设计可基于 bigdata-lab 环境实施 | 使用现有 Hadoop/Hive/Spark/Flink |

## 4 与现有环境的关系

本项目设计基于 `D:\bigdata-lab` 已有技术环境：

| 已有能力 | 本项目利用方式 |
|---------|-------------|
| Hadoop 3.3.6 + Hive 3.1.3 | 数仓存储与查询引擎 |
| Spark 3.5.8 | 离线批处理 ETL 引擎 |
| Flink 1.18.1 | 实时流计算引擎 |
| DataX | 关系数据库离线采集 |
| MySQL 8.0.46 | 元数据存储 + 应用结果存储 |
| Python 3.10.12 | 辅助脚本与数据分析 |

**待新增组件（纳入本项目规划）**：
- Apache Kafka（数据中枢）
- Canal（MySQL Binlog 实时采集）
- DolphinScheduler（调度编排）
- Redis（实时指标缓存）
- HBase（实时明细存储，可选）
