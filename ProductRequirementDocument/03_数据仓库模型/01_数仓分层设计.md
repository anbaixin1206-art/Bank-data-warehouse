## 2 各层详细设计

### 2.1 STG 层 — 临时缓冲层

**职责**：数据采集的原始落地层，不对数据做任何业务转换。

| 属性 | 规范 |
|------|------|
| **存储格式** | Text/CSV（与源系统导出格式一致） |
| **分区策略** | `dt=YYYY-MM-DD`（按采集日期分区） |
| **保留期限** | 3-7天，每日自动清理过期分区 |
| **数据内容** | 原始字段、原始值、原始编码 |
| **不做什么** | 不转换编码、不处理 NULL、不合并字段、不校验 |

**STG 表命名**：`stg_{source_system}_{source_table}`

```sql
-- 示例：STG 层核心银行交易表
CREATE EXTERNAL TABLE stg_core_t_transaction (
    TRANS_ID          STRING,
    ACCOUNT_NO        STRING,
    TRANS_TYPE        STRING,
    TRANS_AMT         STRING,       -- 保留原始字符串，暂不转换
    DR_CR_FLAG        STRING,
    TRANS_TIME        STRING,
    CHANNEL           STRING,
    TELLER_ID         STRING,
    OPP_ACCOUNT       STRING
)
PARTITIONED BY (dt STRING COMMENT '采集日期 yyyy-MM-dd')
ROW FORMAT DELIMITED
FIELDS TERMINATED BY '\t'
STORED AS TEXTFILE
LOCATION '/data/stg/core/t_transaction';
```

**清理策略**：

```sql
-- 每日 00:30 执行：删除 7 天前的 STG 分区
ALTER TABLE stg_core_t_transaction DROP IF EXISTS PARTITION (dt='${date_7_days_ago}');
```

### 2.2 ODS 层 — 操作数据层

**职责**：贴源层，与源系统表结构 1:1 映射，增加采集元数据。

| 属性 | 规范 |
|------|------|
| **存储格式** | ORC + Snappy 压缩 |
| **分区策略** | `dt` + `source_system`（多级分区） |
| **保留期限** | 30天增量 + 每日全量快照 |
| **元数据字段** | 每张表增加 ingest_time, source_system, batch_id |

**ODS 表命名**：`ods_{source_system}_{source_table}`

**ODS 标准元数据字段（每张 ODS 表必须包含）**：

| 字段名 | 类型 | 说明 | 示例 |
|--------|------|------|------|
| `ingest_time` | TIMESTAMP | 数据入库时间 | `2026-06-15 00:30:00` |
| `source_system` | STRING | 来源系统编码 | `CORE` / `PAY` / `LOAN` |
| `batch_id` | STRING | 采集批次号 | `batch_20260615_001` |
| `dt` | STRING | 数据日期分区 | `2026-06-14`（业务日期） |

```sql
-- 示例：ODS 层核心银行交易表
CREATE TABLE ods_core_t_transaction (
    TRANS_ID          STRING    COMMENT '交易流水号',
    ACCOUNT_NO        STRING    COMMENT '账号',
    TRANS_TYPE        STRING    COMMENT '交易类型',
    TRANS_AMT         DECIMAL(18,2) COMMENT '交易金额',
    DR_CR_FLAG        STRING    COMMENT '借贷标志 D-借 C-贷',
    TRANS_TIME        TIMESTAMP COMMENT '交易时间',
    CHANNEL           STRING    COMMENT '渠道编码',
    TELLER_ID         STRING    COMMENT '柜员号',
    OPP_ACCOUNT       STRING    COMMENT '对手账号',
    -- ODS 元数据
    ingest_time       TIMESTAMP COMMENT '数据入库时间',
    source_system     STRING    COMMENT '来源系统 CORE/PAY/LOAN/...',
    batch_id          STRING    COMMENT '采集批次号'
)
PARTITIONED BY (dt STRING COMMENT '业务日期 yyyy-MM-dd')
STORED AS ORC
TBLPROPERTIES ('orc.compress'='SNAPPY');
```

**STG → ODS ETL**：

```sql
-- 每日 01:00 执行
INSERT OVERWRITE TABLE ods_core_t_transaction PARTITION (dt='${yesterday}')
SELECT
    CAST(TRANS_ID AS STRING),
    CAST(ACCOUNT_NO AS STRING),
    CAST(TRANS_TYPE AS STRING),
    CAST(TRANS_AMT AS DECIMAL(18,2)),  -- 类型转换
    CAST(DR_CR_FLAG AS STRING),
    CAST(TRANS_TIME AS TIMESTAMP),
    CAST(CHANNEL AS STRING),
    CAST(TELLER_ID AS STRING),
    CAST(OPP_ACCOUNT AS STRING),
    CURRENT_TIMESTAMP()           AS ingest_time,
    'CORE'                        AS source_system,
    '${batch_id}'                 AS batch_id
FROM stg_core_t_transaction
WHERE dt = '${yesterday}';
```

### 2.3 DWD 层 — 明细数据层

**职责**：数据清洗、标准化、整合，是数仓的核心明细层。⭐ 采用 **Data Vault 2.0 + 传统数仓** 融合建模。

| 属性 | 规范 |
|------|------|
| **存储格式** | ORC + Zlib 压缩（高压缩比） |
| **分区策略** | `dt` 按日期分区 |
| **分桶策略** | 按 HASH_KEY 分桶 (bucket=32)，加速 JOIN |
| **保留期限** | 3年在线 + 归档 |

**DWD 表类型**：

| 表类型 | 命名模式 | 建模方法 | 示例 |
|--------|---------|---------|------|
| **Hub** | `dwd_hub_{entity}` | Data Vault | `dwd_hub_customer` |
| **Link** | `dwd_link_{relation}` | Data Vault | `dwd_link_cust_acct` |
| **Satellite(属性)** | `dwd_sat_{entity}_{content}` | Data Vault + SCD Type 2 | `dwd_sat_customer_info` |
| **Satellite(事实)** | `dwd_sat_{entity}_{content}` | Data Vault (事实型) | `dwd_sat_transaction` |
| **拉链表** | `dwd_zip_{entity}_{content}` | 传统 SCD Type 2 | `dwd_zip_account_status` |
| **事务事实表** | `dwd_fact_{process}` | 传统星型模型 | `dwd_fact_transaction` |

**DWD 表必须包含的元数据字段**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `load_date` | DATE | 数据加载日期 |
| `record_source` | STRING | 数据来源系统 |
| `hash_key` | STRING | MD5 哈希主键 (Hub/Link) |
| `load_end_date` | DATE | 拉链结束日期 (Satellite/拉链表) |

详见 [02_Data_Vault建模规范](./02_Data_Vault建模规范.md)。

### 2.4 DWS 层 — 汇总数据层

**职责**：按主题域进行轻度汇总，构建星型模型（维度表 + 事实表）。

| 属性 | 规范 |
|------|------|
| **存储格式** | ORC + Snappy 压缩 |
| **分区策略** | `dt` + `topic_domain` |
| **分桶策略** | 按维度主键分桶 (bucket=16) |
| **保留期限** | 5年在线 + 归档 |

**DWS 表类型**：

| 表类型 | 命名模式 | 说明 |
|--------|---------|------|
| **维度表** | `dim_{entity}` | 缓慢变化维度 (SCD Type 2) |
| **日快照事实** | `dws_{domain}_daily_{metric}` | 日粒度汇总 |
| **累计快照事实** | `dws_{domain}_acc_{metric}` | 累计至今汇总 |
| **多粒度事实** | `dws_{domain}_{粒度}_{metric}` | 周/月/季粒度 |

**维度表标准字段**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `{dim}_key` | BIGINT | 维度代理键 (自增) |
| `{dim}_hash_key` | STRING | 对应 DWD Hub 的 HASH_KEY |
| `{dim}_natural_key` | STRING | 业务自然键 |
| `effective_date` | DATE | 生效日期 |
| `expiry_date` | DATE | 失效日期 (9999-12-31 表示当前) |
| `is_current` | BOOLEAN | 是否当前记录 |

**汇总事实表标准字段**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `dt` | STRING | 快照日期 |
| `{dim}_key` | BIGINT | 维度外键 |
| `{metric}_cnt` | BIGINT | 计数指标 |
| `{metric}_amt` | DECIMAL(18,2) | 金额指标 |
| `{metric}_qty` | BIGINT | 数量指标 |
| `etl_time` | TIMESTAMP | ETL 处理时间 |

### 2.5 ADS 层 — 应用数据层

**职责**：面向具体应用的高度汇总表、大宽表、数据集市。

| 属性 | 规范 |
|------|------|
| **存储格式** | ORC/Parquet + Snappy |
| **分区策略** | `dt` + `app_name` |
| **保留期限** | 按应用需求（监管报表永久保留） |

**ADS 表类型**：

| 表类型 | 命名模式 | 消费应用 |
|--------|---------|---------|
| **管理驾驶舱** | `ads_mgmt_{kpi_name}` | 大屏/BI |
| **监管报送** | `ads_reg_{report_type}_{table}` | 监管机构 |
| **客户画像** | `ads_cust_{scene}` | CRM/营销 |
| **风险管理** | `ads_risk_{scene}` | 风控系统 |
| **自助BI** | `ads_bi_{domain}_wide` | BI 分析工具 |

## 3 跨层数据流转规范

### 3.1 数据流向

```
源系统 → STG → ODS → DWD → DWS → ADS → 数据应用
                       ↘ DWD → DWS → ADS (实时链路)
```

### 3.2 各层禁止事项

| 层 | 禁止 | 正确做法 |
|-----|------|---------|
| **STG** | 禁止做任何类型转换 | 保持源系统原始格式 |
| **ODS** | 禁止跨表 JOIN | 只做类型转换和元数据添加 |
| **DWD** | 禁止直接对 ODS 做复杂聚合 | 聚合在 DWS 层 |
| **DWS** | 禁止直接引用 STG/ODS 表 | 只能引用 DWD 层表 |
| **ADS** | 禁止跨 ADS 表 JOIN | 只做最终展示逻辑 |
| **全层** | 禁止反向依赖（下层依赖上层） | 严格单向依赖 |

### 3.3 ETL 幂等性要求

所有层的 ETL 必须支持 **覆盖重跑**（`INSERT OVERWRITE` 分区），保证指定日期分区的数据可重复生成，结果一致。

```sql
-- 正确：覆盖写入指定分区
INSERT OVERWRITE TABLE dws_dep_daily_bal PARTITION (dt='${yesterday}')
SELECT ...;

-- 错误：追加写入（重跑会导致重复数据）
INSERT INTO TABLE dws_dep_daily_bal PARTITION (dt='${yesterday}')
SELECT ...;
```

## 4 索引与性能优化

### 4.1 分桶 (Bucketing)

| 层 | 分桶键 | 桶数 | 目的 |
|-----|--------|------|------|
| DWD Hub | `hash_key` | 32 | 加速 Hub JOIN |
| DWD Link | `hash_key` | 32 | 加速 Link-Hub JOIN |
| DWD Sat | `hash_key` | 32 | 加速 Sat-Hub JOIN |
| DWS 事实 | 主维度外键 | 16 | 加速维度-事实 JOIN |

### 4.2 分区裁剪

```sql
-- 查询优化：利用分区裁剪
SET hive.partition.pruning = strict;

-- 查询自动裁剪非必要分区
SELECT * FROM dwd_sat_transaction
WHERE dt BETWEEN '2026-06-01' AND '2026-06-14'
  AND hash_key = 'abc123...';  -- 同时利用分桶裁剪
```

### 4.3 统计信息

```sql
-- 定期更新统计信息（DWS 层每日，DWD 层每周）
ANALYZE TABLE dws_dep_daily_bal PARTITION (dt='${yesterday}') COMPUTE STATISTICS;
ANALYZE TABLE dws_dep_daily_bal PARTITION (dt='${yesterday}') COMPUTE STATISTICS FOR COLUMNS;
```
