## 1 实时处理三大场景

| 场景 | 延迟要求 | 引擎 | 输出 |
|------|---------|------|------|
| **A. 实时风控** | < 500ms | Flink CEP | 告警/阻断 |
| **B. 实时对账** | < 5s | Flink SQL | 差异报表(MySQL) |
| **C. 实时大屏** | < 10s | Flink SQL | 指标(Redis) → Web |

## 2 实时数据流架构

```
MySQL源库 ──Canal──► Kafka ──Flink──┬── Redis (实时指标, TTL=10s)
                    bank.ods.*      ├── MySQL (对账结果)
                                    ├── HBase (告警留痕)
                                    └── Alert (钉钉/邮件)
```

## 3 Kafka中枢设计

Kafka同时服务于实时和离线，解决Lambda双轨不一致问题：

| 数据链路 | 消费模式 | Offset |
|---------|---------|--------|
| Flink实时 | latest | 从最新开始 |
| Spark离线 | earliest (昨日00:00-23:59) | 按时间戳定位 |

```
Kafka Topic: bank.ods.{system}.{table}
  实时消费 ──→ Flink CEP/SQL ──→ 实时输出
  离线回放 ──→ Spark (T+1) ──→ Hive ODS
```

---

## 4 Lambda架构数据一致保证

- 实时和离线使用**同一Kafka Topic**作为数据源
- Flink实时写入DWD当前分区，Spark T+1覆盖写入同日分区
- 最终以离线数据为准，实时数据用于时效性要求高的场景
